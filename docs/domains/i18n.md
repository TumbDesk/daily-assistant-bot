# Internationalization (i18n)

The bot supports German and English UI strings. Commands use short English and German aliases (e.g. `/event` or `/termin`); message text follows the resolved locale.

## Key components

| File | Responsibility |
|------|----------------|
| `locales/en.json` | English strings |
| `locales/de.json` | German strings |
| `services/locale_service.py` | Locale resolution for handlers and users |
| `services/i18n_util.py` | `t()`, `t_list()`, `LocalizedError` |
| `services/bot_commands.py` | Telegram menu commands from `menu_commands` |
| `views/message_factory.py` | Formatted messages using locale keys |

## Locale resolution

Order of precedence (see `locale_service`):

1. User setting from `/settings` (`User.locale`) if set
2. Telegram `language_code` from the user's client
3. Default: English (`MessageFactory.DEFAULT_LOCALE`)

The `restricted` decorator stores resolved locale in `context.user_data["locale"]` for handlers.

## Translation files

JSON key → string value. Keys use snake_case (e.g. `agenda_events_today_header`).

Placeholders use `{name}` style and are filled via `t()` or `MessageFactory` methods.

`locales/` is mounted as a volume in Docker so strings can be edited without rebuilding the image.

### Telegram menu commands

Each locale file may define a `menu_commands` object: command name (without `/`) → short description. On bot startup, `setup_bot_commands()` in `services/bot_commands.py` reads `locales/de.json` and `locales/en.json` and registers the entries via `set_my_commands()` with `language_code="de"` and `language_code="en"` respectively.

## LocalizedError

Service layer raises `LocalizedError("err_key")` with optional kwargs. Handlers catch and display via `MessageFactory` or `t()` with the user's locale.

Keeps business logic free of hard-coded user-facing strings.

## Adding a language

1. Copy `locales/en.json` to `locales/<code>.json`
2. Translate all keys
3. Register the locale code in locale resolution logic if not auto-detected
4. Add settings keyboard option if users should pick it in `/settings`

## Static vs dynamic text

- **Static** — all user-visible phrases in `locales/*.json`
- **Dynamic** — dates/times formatted in services using `timezone_util` and locale-aware labels where applicable

Event recurrence labels in `calendar_service.WEEKDAY_LABELS` are currently German-only in code; English agenda uses locale keys from JSON for most output.
